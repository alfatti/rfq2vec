# Training a Transaction Foundation Model on the RFQ Simulator

A strategy note, written against the NVIDIA "transaction foundation model" blueprint
and the `rfqsim` output schema. The organizing claim: because we own the
data-generating process, the simulator's job is not to be the product but to be the
*instrument* that tells us how large the product needs to be and exactly which
structures each size will and will not learn.

---

## 1. The starting point is better than the blueprint's

The NVIDIA example spends most of its effort inventing a token out of raw card-swipe
fields. It hashes merchant names into 2,000 buckets, cuts dollar amounts at seven
fixed thresholds, maps merchant-category codes through a lookup, and emits each
transaction as a block of twelve field-tokens (`AMT_3 MERCH_1498 CAT_RETAIL ...`).
A customer's history becomes one sentence: `<bos> txn1 <sep> txn2 <sep> ... <eos>`,
chunked at 315 transactions to fit a 4,096-token window. The model is a ~29M-parameter
Llama decoder (hidden 512, 8 layers, grouped-query attention, RoPE, SwiGLU, vocab
6,251) trained with plain next-token prediction, and evaluated by pulling the last
token's hidden state as a 512-dim embedding and feeding it to XGBoost for fraud.

We are ahead of that on two fronts that matter.

**We already emit the atom.** `rfq_lines.token_id` is the (CUSIP, side) unit that the
whole Arora analysis established as correct — because the unsigned CUSIP collapses the
buy and sell senses into a frequency-weighted mixture that destroys the substitution
geometry. We do not need to reverse-engineer a token; the embedding row for that token
*is* the Arora word vector, and the model's core competency is to learn its geometry.

**We own the latents and the probabilities.** We emit the ground-truth walk and regime
path (`context_grid`, `regime_path`) and, in `event_truth`, the exact masked log
partition function, the chosen logit, and `log_p_chosen` — the true generative
probability of every realized choice. No real-data foundation model can grade itself
this way. It is the reason to train on the simulator at all.

### What the model tokenizes from

The canonical line (`rfq_lines`, 36 columns) carries, per RFQ leg: the client
(`client_id`, the "author"), the (CUSIP, side) atom (`token_id`), disclosed vs true
side, size (`qty_par`, `qty_bucket`), competition (`n_dealers`), package structure
(`package_id`, `line_no`, `n_lines`, `package_type`), timing (`ts`, `grid_idx`), and
outcomes (`enquiry_outcome`, `our_result`, spreads). The policy-gated recommender
columns (`rec_token_id`, `rec_delta`, `rec_propensity`, `policy_id`) sit alongside for
downstream bandit work but are held out of pretraining.

## 2. Tokenization

### 2.1 Anchored-composite blocks, not field-shatter

Two extremes, and we want neither in its pure form:

- **Field-shatter (NVIDIA)**: break every line into many field-tokens. Rich
  within-line prediction, but it dilutes the one thing our theory is built around — the
  (CUSIP, side) geometry — by surrounding each anchor with scaffolding that competes
  for the model's capacity and for its next-token loss budget.
- **Pure item stream**: each line is exactly one (CUSIP, side) token. Cleanest possible
  map to RAND-WALK, but it discards the structured covariates (client, size, outcome,
  timing) that make this a *transaction* model rather than an item2vec.

The compromise keeps the atom as a single first-class vocabulary entry and wraps it in
a **short, fixed** block so the anchor stays dominant. Per leg:

```
TDLT_*   log-binned time gap since this client's previous RFQ   (the intensity channel)
CLI_*    client id (the author; carries the u_k tilt)
TOK_*    the (CUSIP, side) atom  ← the anchor; its embedding = the Arora word vector
SZ_*     log-binned par size bucket
NDLR_*   dealer-count bucket (competition)
OUT_*    enquiry_outcome ⊗ our_result (the predictable label channel)
<sep>
```

Roughly seven tokens per line, one of which is the anchor. Multi-leg packages (switch,
list) emit consecutive `TOK` sub-blocks inside one event under a lighter delimiter, so
the model sees the switch as an opposite-side pair and the list as one-way legs —
strong local grammar that later serves as a capacity probe (Test F).

Two deliberate departures from the blueprint, both forced by *our* DGP:

- **Time is a first-class token, via the gap, not the wall-clock field.** NVIDIA bins
  HOUR/DOW/MONTH. Our clustering is generated by a Markov-modulated Poisson intensity,
  so the informative quantity is the *inter-arrival gap*. `TDLT` is the only channel
  through which the model can see the regime/intensity structure at all. This matters
  for Test D and for a clean negative control: strip `TDLT` and no model, at any size,
  can beat a constant-rate baseline on timing — a direct consequence of the
  partition-function-concentration result (the item-selection softmax has no
  aggregate-activity channel).

- **The embedding layer mirrors the generative token model.** In the simulator a bond
  vector is `v_w(t) = s_w(t) · normalize(B · x_w(t) + ε_w)`: norms carry frequency,
  directions carry meaning, and the direction is an attribute map `B` applied to
  observable bond features plus a small residual — a fastText-style decomposition. The
  model's `TOK` embedding should be built the same way: a learned linear map over the
  bond's observable features (sector, rating, tenor/roll-down, issue size, age) plus a
  per-token residual, rather than 90,000 free rows. This controls parameter count (see
  §3.3) and makes the model's inductive bias match the DGP — and it keeps the residual
  assumption falsifiable in exactly the way the simulator's stability gate does.

### 2.2 Two sequence axes — and why we train on both

There are two legitimate "sentences" in RFQ data, and they teach different things.

- **Client-session sequences.** All of one client's RFQs in time order. The direct
  analogue of the cardholder history: the model learns each client's next-bond
  distribution, activity rhythm, and personal substitution pattern; the tilt `u_k`
  becomes learnable as a per-client offset. This is the personalization workhorse and
  the bulk of the signal.

- **Market-tape windows.** All clients interleaved in time order, each block prefixed by
  its `CLI` token. This is the structure the theory calls the *productive advantage over
  single-author NLP corpora*: the interleaving lets cross-client co-occurrence recover
  the shared market geometry, while same-client excess co-occurrence recovers the
  clientele covariance `Σ_u` separately. That separation only exists if the model sees
  many authors woven onto one tape.

**Proposal: a dual-corpus curriculum** — pretrain mostly on client-session sequences,
mix in a minority of market-tape windows. The session view gives the personalization
the product needs; the tape view is what makes the cross-client geometry and `Σ_u`
recoverable (and testable — Test C). This is a real fork, flagged in §5.

### 2.3 Vocabulary accounting

`TOK` dominates: three senses (buy/sell/undisclosed) per instrument. At the 30k-instrument
investment-grade target that is ~90,000 anchor tokens; add clients (`CLI`, a few thousand
to tens of thousands), plus a few hundred bucket tokens for `TDLT`/`SZ`/`NDLR`/`OUT` and
the specials. Call it ~100k vocabulary — versus NVIDIA's 6,251, precisely because they
hash and bucket their high-cardinality fields and we refuse to hash the thing whose
geometry we care about. The attribute-factorized embedding of §2.1 is what makes a
100k vocabulary affordable.

## 3. Data volume and model size

### 3.1 Token budget

From the production run (501,926 lines over 10 full-market days ≈ 50k lines/day) and
~7 tokens/line:

| Horizon | Lines | Tokens | Chinchilla-optimal params (20 tok/param) |
|---|---|---|---|
| 1 year (252d) | 12.6M | ~88M | ~4M |
| 5 years | 63M | ~440M | ~22M |
| 10 years | 126M | ~880M | ~44M |
| 10y × 4 universe seeds | 504M | ~3.5B | ~180M |
| 10y × 16 seeds | 2.0B | ~14B | ~700M |

Two readings jump out. First, **the NVIDIA 29M-parameter model is Chinchilla-matched to
roughly one desk, a few years of RFQ data** — a useful anchor for "what size is the
obvious baseline." Second, we can manufacture as many tokens as we like: more
instruments, more clients, longer horizons, or — the cleanest axis — multiple
independent universe draws (`build_universe` is a pure function of seed and dials). One
to ten billion tokens is comfortably in reach.

### 3.2 The synthetic-data twist: an entropy floor, not a memorization tail

Chinchilla's 20:1 rule minimizes next-token loss on the training distribution. Two
things make our situation different, and both push the sizing logic off the naive
curve.

**There is no long tail to memorize.** Real foundation models keep improving with data
partly because the world has endless idiosyncratic facts. Our tokens are draws from a
fixed DGP; past the point where the model has absorbed that process, more tokens only
re-estimate the same distribution. So the *data* axis saturates at the DGP's intrinsic
entropy, which — uniquely — we can compute from `event_truth`. We should plan to
generate to the point of diminishing returns against that known floor, not beyond.

**Minimal loss is not our objective; recovering latent structure is.** A model can sit
right on the Chinchilla-optimal perplexity frontier and still fail to recover, say, the
full client covariance, because that structure lives in the tails and contributes almost
nothing to *average* next-token loss. When the target structure is high-dimensional but
low-loss-contribution, the right move is to size the model *above* the Chinchilla point
for its token count and let the extra capacity resolve structure that perplexity alone
would never pay for. Which structures those are is exactly §4.

### 3.3 The embedding table can dwarf the transformer

At a 90k–100k vocabulary, a naive embedding table is enormous relative to NVIDIA's
whole model:

| Vocab | Hidden 256 | Hidden 512 | Hidden 1024 |
|---|---|---|---|
| 11k (HY, today) | 2.8M | 5.6M | 11.3M |
| 90k (IG target) | 23M | 46M | 92M |
| 100k | 26M | 51M | 102M |

At hidden 512 the IG embedding table alone (46M) exceeds NVIDIA's entire 29M model.
This is the structural reason the attribute-factorized embedding of §2.1 is not a nicety
but a requirement: it replaces most of those free rows with a small learned map over
bond features plus a residual, so parameters land in the transformer (where the
sequential regime/geometry reasoning happens) rather than in a lookup table.

### 3.4 A three-rung model ladder

Sized to *what structure we want back*, not only to token count:

- **Small — recovers the easy rungs.** ~5–15M transformer params (hidden 256, 6 layers)
  over factorized embeddings; Chinchilla-matched to ~0.2–0.5B tokens (1–5 years, one
  market). Recovers marginal item popularity (the Zipf norm law), first-order
  substitution (nearest-neighbour bonds within a sector), and a client's own recency.

- **Medium — the NVIDIA-scale baseline.** ~30M params (hidden 512, 8 layers), ~0.6–1B
  tokens. Recovers full within-sector substitution geometry, buy/sell sense separation,
  size-conditioned behaviour, and basic regime awareness through `TDLT`.

- **Large — recovers the hard rungs.** ~150–350M params (hidden 1024, 16–24 layers),
  fully attribute-factorized embeddings, 3–10B tokens from multiple universe seeds and
  decade horizons. Recovers the full-rank cross-client geometry, client-*conditional*
  substitution (the `u_k` tilt as a context-dependent rotation, not a per-client bias),
  the regime-switching intensity coupling, the clientele covariance `Σ_u`, and slow
  roll-down/aging over long contexts.

Keep NVIDIA's architectural choices throughout (RoPE with a high base for long-context
extrapolation, grouped-query attention, SwiGLU, RMSNorm, no dropout); they are sound and
architecture is not where our leverage is. Our leverage is the token, the embedding
factorization, and the evaluation.

## 4. Tests: what a large model recovers that a small one cannot

This is the centrepiece, and it is where owning the DGP pays off. The general principle:
a capability needs capacity when its target is **high-dimensional, conditional, or
compositional**, so that a small model is forced to collapse it to a marginal or
low-rank approximation. We built several such structures into the simulator on purpose,
and we have ground-truth latents to grade against.

The organizing thesis: **the analytic identification ladder predicts the empirical
capability ladder.** We derived, in order, that norms are identified by frequencies
(easy), directions by cross-client co-occurrence (medium), `Σ_u` by same-client excess
co-occurrence (hard), and regimes by the arrival structure (hard, and on a separate
channel). A model should recover those rungs in the *same order*, with the harder rungs
switching on only at larger scale. Each test below names a DGP fact, the probe, the
metric, and — the part that makes it a *scale* test — the mechanism that forces capacity.

**A. Full-rank substitution geometry.** *DGP fact:* we chose the random-walk generator
specifically because its PMI Gram matrix is full-rank and positive semidefinite, unlike
the CTMC block structure (rank-R), and the word vectors live in d=64 dimensions.
*Probe:* extract `TOK` embeddings, form the pairwise Gram matrix, and align its top-k
subspace to the ground-truth `⟨v_w, v_w'⟩` via principal angles, sweeping k. *Metric:*
subspace alignment vs k. *Mechanism:* a small model (hidden < d, or too few tokens to
estimate tail singular values) recovers only the leading directions — the dominant
sectors and themes — and its geometry stays low-rank; alignment tracks for small k then
diverges. A large model tracks to high k. This is the block-vs-full-rank argument turned
into a capacity probe.

**B. Side-sense polysemy.** *DGP fact:* (CUSIP, side) is the atom because the unsigned
CUSIP collapses buy and sell into a frequency-weighted superposition (Arora's polysemy
result), and two-way is a genuine third sense. *Probe:* do buy and sell embeddings for
the same instrument separate, does the two-way token sit where the predicted
superposition says it should, and can the model complete a switch with the correct
*opposite* side? *Metric:* buy/sell separation margin relative to the instrument
centroid; switch-completion accuracy. *Mechanism:* under vocabulary pressure a small
model ties buy and sell toward their shared instrument centroid — they share context, so
keeping them apart costs capacity that barely moves average loss — effectively
re-collapsing the polysemy. A large model holds the separation.

**C. Client-conditional substitution.** *DGP fact:* selection is `∝ exp⟨v_w, c_t + u_k⟩`,
so the client tilt `u_k` *rotates* the substitution pattern — two clients in the same
market state have different next-bond distributions. This is a client×item interaction,
not an additive popularity term. *Probe:* hold market context fixed, swap the `CLI`
token, and check the predicted-bond shift matches the DGP's `exp⟨v, Δu⟩`; regress the
model's per-client logit tilt on the true `u_k`. *Metric:* recovery correlation of
`u_k`; fraction of client×item interaction variance explained. *Mechanism:* a small
model captures marginal client popularity (client k trades more X overall) but not the
conditional rotation (client k substitutes differently *given the current theme*),
because the interaction is a bilinear form `v_wᵀ u_k` over all (w, k) and a small model
resolves only a few client "types." This is the model-side image of "same-client excess
co-occurrence identifies the clientele covariance."

**D. Regime and intensity structure.** *DGP fact:* the Markov-modulated Poisson layer
drives aggregate activity while the walk drives which bonds, and the two are structurally
separate; regimes modulate walk drift/diffusion/temperature and the arrival rate, with
the stress state emergent. *Probe:* (1) predict the next inter-arrival bucket from recent
`TDLT`+`TOK` tokens against a constant-rate baseline; (2) test whether the "how busy"
subspace (predicts next `TDLT`) is disentangled from the "which theme" subspace (predicts
next `TOK`); (3) inject a regime change from the known `regime_path` and measure
change-point response latency. *Metric:* inter-arrival predictive log-likelihood lift over
homogeneous Poisson; change-point latency. *Mechanism:* a Markov-modulated intensity
requires the model to infer and *track a hidden state* across the sequence — genuinely
sequential and higher-capacity than static next-item. A small model collapses to a
time-averaged rate; a large one tracks the switching. And there is a theoretically exact
ceiling: with `TDLT` removed, no size beats the marginal, because the item softmax has no
aggregate-activity channel. That negative control isolates the channel cleanly.

**E. Roll-down and aging.** *DGP fact:* bonds carry roll-down through time-varying
attributes `x_w(t)` (tenor shortens, benchmark boosts fade) and an age-decay norm
schedule `s_w(t)` — slow effects visible only over long spans. *Probe:* does the model's
implied popularity and geometry for a bond drift correctly as it ages across a long
context? *Metric:* correlation of implied drift with the true attribute schedule.
*Mechanism:* short-context or small models see only local snapshots and treat each bond
as static; large long-context models (this is where the RoPE window and extrapolation
earn their place) recover the trajectory. Context length is a capacity axis here as much
as parameter count.

**F. Package grammar (composition).** *DGP fact:* a switch is an opposite-side pair on two
different instruments, a list is one-way legs, and package temperature is sharpened.
*Probe:* switch-completion (given leg 1 = buy X, predict leg 2 = sell of a *substitute*
Y) and list-continuation. *Metric:* opposite-side accuracy and substitute quality (cosine
to the true leg-2 in DGP geometry). *Mechanism:* the rule requires *binding* two facts —
"this is a switch" (from package context) and "opposite side of a substitute" (from
geometry). Binding is where attention depth and capacity classically matter; a shallow or
small model gets a plausible bond (the marginal) but not the bound completion.

**G. Calibration against the true generative probability — the punchline.** *DGP fact:*
`event_truth` stores the exact `log_p_chosen` for every realized choice. *Probe:* compare
the model's predicted `log p(next token | history)` to the DGP's true value on held-out
lines. *Metric:* excess KL above the DGP's intrinsic entropy floor. *Mechanism and why it
matters:* this is not a proxy — it is the actual generative probability, so we can plot a
scaling curve of excess-KL-above-floor versus model size and show small models plateauing
above the floor (irreducible approximation error) while large models approach it. **Real
foundation models can never draw this curve, because they do not know their floor.** It is
the single most compelling artifact the simulator makes possible, and it is the honest way
to answer "how big does the real model need to be": big enough that its excess KL against
the simulated floor has flattened.

## 5. Critical choices to confirm before building

1. **Sequence axis / curriculum (§2.2).** Client-session only, market-tape only, or the
   dual-corpus mix? The mix is my recommendation because it is the only configuration in
   which the cross-client geometry and `Σ_u` are recoverable — but it costs more
   engineering and the mixing ratio is a knob. This is the biggest fork.

2. **Composite anchor vs field-shatter (§2.1).** I recommend the anchored-composite block
   (keep (CUSIP, side) as one vocabulary entry). Field-shatter is closer to the blueprint
   and simpler to lift, but dilutes the geometry the theory is built on.

3. **Attribute-factorized embedding vs free rows (§2.1, §3.3).** I recommend factorized,
   both to afford a 100k vocabulary and to match the DGP's own token model. The cost is a
   more custom embedding layer than a stock HuggingFace `nn.Embedding`.

4. **How much data to manufacture (§3.2).** Generating to the DGP entropy floor is
   principled but needs the floor measured first (cheap — it is already in `event_truth`).
   Worth deciding whether we target the floor or a fixed compute budget.

## 6. Build plan (phased)

1. **Corpus adapter.** `rfq_lines` → token blocks → session and tape corpora, reusing the
   existing enum registries so token semantics stay single-sourced. Emit alongside the
   `event_truth` join keys so every training line can be matched back to its true
   `log_p_chosen` for Test G.
2. **Factorized tokenizer + embedding.** The attribute map `B` over bond features plus
   residual, mirroring `token_vectors` in the simulator.
3. **Three model configs** (small/medium/large) on the NVIDIA architecture spine, trained
   with next-token prediction; the large one sharded across the H200s the simulator
   already targets.
4. **The test harness (§4).** Each probe reads the ground-truth latents
   (`context_grid`, `regime_path`, universe vectors) and `event_truth`; the deliverable is
   a scaling table — every metric as a function of model size — with Test G's
   excess-KL-vs-floor curve as the headline.

The endpoint is not just a trained model. It is a scaling curve, measured against a known
floor, that tells us how large the *real*-data foundation model has to be to recover each
structure we care about — the question the blueprint cannot pose because it never knows
the process that made its data.
